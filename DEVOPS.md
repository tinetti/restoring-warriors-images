# Restoring Warriors Images Deployment Architecture

This document describes how `restoring-warriors-images` should be managed by the `devops-tools` repository. It is intentionally an architecture and operating contract, not an implementation patch list.

## Decision summary

- **The application repository owns the application image contract**: source code, `Dockerfile`, local-only `docker-compose.yml`, and a small workflow that asks this repo to deploy a commit.
- **This repository owns deployment orchestration**: app inventory, infrastructure, Docker image build/push, droplet provisioning, generated runtime Compose files, Nginx routing, SSL, backups, and deployment workflows.
- **Do not publish or commit a production `docker-compose.yml` from the app repo.** The production/runtime Compose file should be generated by this repo from `docs/apps.yaml` and rendered onto the target droplet at `/home/appuser/docker-compose/<app-name>/docker-compose.yml`.
- **The app repo may keep its existing `docker-compose.yml` for local development only.** It should not be treated as the source of truth for production ports, secrets, restart policy, networks, or image tags.

## Target deployment model

```text
restoring-warriors-images repo
  ├─ Dockerfile                 # builds the deployable MinIO image
  ├─ docker-compose.yml          # local developer convenience only
  └─ GitHub Actions              # builds image, then dispatches devops-tools workflow
          │ workflow_dispatch(app_name, tag)
          ▼
devops-tools repo
  ├─ docs/apps.yaml              # source of truth for app/environment registration
  ├─ .github/workflows/deploy-*.yml
  ├─ .github/scripts/docker-build.sh
  ├─ scripts/deploy.sh           # runs on target droplet
  ├─ ansible/                    # provisions droplets, Docker, Nginx, SSL, scripts
  └─ infra/                      # DigitalOcean droplets/firewall/networking
          │
          ├─ builds docker.io/<docker-user>/restoring-warriors-images:<tag>
          ├─ pushes image to Docker Hub
          ├─ SSH deploys to dev/prod app droplet
          └─ proxy routes https://<domain> to the app container
```

## Application registration

Add one entry per app/environment to `docs/apps.yaml`.

Recommended initial names:

```yaml
apps:
  - name: restoring-warriors-images
    domain: test-minio.restoringwarriors.com
    repo: tinetti/restoring-warriors-images
    branch: main
    port: 9000
    droplet: dev
    env:
      MINIO_ROOT_USER: "__SET_ON_DROPLET_OR_SECRET_STORE__"
      MINIO_ROOT_PASSWORD: "__SET_ON_DROPLET_OR_SECRET_STORE__"
      MINIO_BUCKET: restoring-warriors-images

  - name: restoring-warriors-images-prod
    domain: minio.restoringwarriors.com
    repo: tinetti/restoring-warriors-images
    branch: main
    port: 9000
    droplet: prod
    env:
      MINIO_ROOT_USER: "__SET_ON_DROPLET_OR_SECRET_STORE__"
      MINIO_ROOT_PASSWORD: "__SET_ON_DROPLET_OR_SECRET_STORE__"
      MINIO_BUCKET: restoring-warriors-images
```

Notes:

- `name` becomes the Docker image name and Compose service/project identity. Keep it stable.
- `repo` should be `tinetti/restoring-warriors-images`.
- `port` should be `9000` because the MinIO container exposes `9000` (API) and `9001` (console). The Nginx proxy should route to `9000`.
- `MINIO_ROOT_USER` and `MINIO_ROOT_PASSWORD` must not be committed to either repo. Store them in a secret store or droplet-local `.env` file.

## Docker image contract

The app repo owns the Docker image contract:

- `Dockerfile` pins the MinIO image tag (e.g., `quay.io/minio/minio:RELEASE.2025-09-07T16-13-09Z-cpuv1`).
- The container exposes port `9000` (API) and `9001` (console).
- The container listens on `0.0.0.0`.

Expected image naming:

```text
docker.io/<DOCKER_USERNAME>/restoring-warriors-images:<tag>
docker.io/<DOCKER_USERNAME>/restoring-warriors-images-prod:<tag>
```

If we do not want separate dev/prod image names, update the tooling to support a separate `image_name` field in `docs/apps.yaml`; otherwise the current convention uses `app.name` as the image name.

## Compose ownership

### Local Compose in the app repo

`docker-compose.yml` is for local development and manual smoke tests:

```bash
docker compose up --build
```

It can reference `.env` values and local defaults. It should not encode production deployment policy.

### Runtime Compose on droplets

Runtime Compose files should be generated by this repo, not copied from the app repo.

Target path on each app droplet:

```text
/home/appuser/docker-compose/restoring-warriors-images/docker-compose.yml
/home/appuser/docker-compose/restoring-warriors-images-prod/docker-compose.yml
```

The generated runtime Compose should include:

- immutable image reference for the requested tag;
- `restart: unless-stopped`;
- host/container port mapping from `docs/apps.yaml`;
- production environment variables from a safe source;
- any required Docker networks/volumes;
- healthcheck configuration using MinIO's built-in health endpoint (`/minio/health/live`).

The source of truth for the generated file should be one of these, in order of preference:

1. `ansible/roles/app-docker-compose/templates/docker-compose.yml.j2` for provision-time generation; and/or
2. `scripts/deploy.sh` for first-deploy fallback generation.

Avoid having multiple divergent Compose templates. If both remain, keep them behaviorally identical or make one call the other.

## Secrets and environment variables

The app needs MinIO credentials:

```text
MINIO_ROOT_USER
MINIO_ROOT_PASSWORD
```

Recommended policy:

- `MINIO_ROOT_USER` and `MINIO_ROOT_PASSWORD` must not be committed to either repo.
- Store deploy-time secrets in GitHub environment secrets, the droplet environment, Docker secrets, or a generated `.env` file on the droplet with mode `0600`.
- This repo's local operator secrets should continue to come from `scripts/load-secrets.sh` and macOS Keychain when running Terraform, Ansible, or deployment commands locally.

Recommended future runtime layout on the droplet:

```text
/home/appuser/docker-compose/restoring-warriors-images/
  ├─ docker-compose.yml          # generated, non-secret
  └─ .env                        # generated or manually installed, secret, mode 0600
```

The Compose file can then use:

```yaml
env_file:
  - .env
environment:
  MINIO_BUCKET: restoring-warriors-images
```

## Required devops-tools changes before this app is fully managed

The current repo has most of the intended pieces, but these gaps should be resolved as part of onboarding `restoring-warriors-images`:

1. **Register the app in `docs/apps.yaml`.**
   - Add dev and prod entries.
   - Use environment-specific domains.

2. **Ensure Ansible loads `docs/apps.yaml` into the `apps` variable.**
   - `ansible/roles/app-docker-compose` and `ansible/roles/nginx-proxy` expect an `apps` variable.

3. **Fix proxy routing to app droplets.**
   - `ansible/roles/nginx-proxy/templates/nginx.conf.j2` currently proxies to `127.0.0.1:<port>`.
   - With separate proxy and app droplets, Nginx should route to the target app droplet private IP, not proxy-local localhost.

4. **Make image tags commit-specific.**
   - App repo dispatch should pass a commit SHA for dev deploys and a release tag for prod deploys.
   - `.github/scripts/docker-build.sh` should checkout the exact SHA/tag before building.

5. **Provide runtime secrets to the droplet.**
   - Decide whether secrets are delivered by GitHub Actions over SSH, Ansible from Keychain, Docker secrets, or a droplet-local `.env` file.

6. **Ensure `DOCKER_USERNAME` exists on the droplet or is rendered into deploy scripts/Compose.**

## App repo workflow contract

Add a workflow in `restoring-warriors-images` that:

1. builds and validates the Docker image;
2. dispatches this repo's dev deployment workflow on `main` pushes;
3. dispatches this repo's prod deployment workflow on releases/tags.

Example dev trigger shape:

```yaml
name: Request dev deploy

on:
  push:
    branches: [main]

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: docker/setup-buildx-action@v3
      - name: Build Docker image
        uses: docker/build-push-action@v5
        with:
          context: .
          push: false
          cache-from: type=gha
      - name: Dispatch devops deploy
        env:
          INFRA_REPO: tinetti/devops-tools
          GITHUB_TOKEN: ${{ secrets.DEVOPS_WORKFLOW_TOKEN }}
        run: |
          curl -fsS -X POST \
            -H "Authorization: Bearer $GITHUB_TOKEN" \
            -H "Accept: application/vnd.github+json" \
            https://api.github.com/repos/${INFRA_REPO}/actions/workflows/deploy-dev.yml/dispatches \
            -d "{\"ref\":\"main\",\"inputs\":{\"app_name\":\"restoring-warriors-images\",\"tag\":\"${GITHUB_SHA}\"}}"
```

The app repo needs a token that can dispatch workflows in this repo. Prefer a fine-scoped GitHub App token or fine-grained PAT over a broad classic token.

## Deployment flow

### Development

```text
push to restoring-warriors-images main
  -> app repo CI: build Docker image
  -> dispatch devops-tools deploy-dev.yml(app_name=restoring-warriors-images, tag=<sha>)
  -> devops-tools checks docs/apps.yaml
  -> clone tinetti/restoring-warriors-images at <sha>
  -> docker build/push docker.io/<user>/restoring-warriors-images:<sha>
  -> SSH to dev app droplet
  -> update generated Compose image tag
  -> docker compose pull/up
   -> validate http://test-minio.restoringwarriors.com/minio/health/live
```

### Production

```text
create release/tag in restoring-warriors-images
  -> app repo CI: build Docker image
  -> dispatch or manually run devops-tools deploy-prod.yml(app_name=restoring-warriors-images-prod, tag=<release-tag>)
  -> build/push docker.io/<user>/restoring-warriors-images-prod:<release-tag>
  -> SSH to prod app droplet
  -> update generated Compose image tag
  -> docker compose pull/up
   -> validate http://minio.restoringwarriors.com/minio/health/live
```

## DNS and SSL

- Point the chosen domains at the proxy droplet public IP.
- Run the SSL setup from this repo after app registration and DNS propagation.
- Nginx config should be generated from `docs/apps.yaml` and the app droplet private IP map.

Suggested commands after implementation:

```bash
just test-apps-yaml
just docker-ansible -- --syntax-check
just ssl-auto
```

Per repo policy, before Terraform, Ansible, or deployment commands, load secrets with:

```bash
source scripts/load-secrets.sh
export DO_TOKEN=$(retrieve_secret DO_TOKEN)
export SSH_PUBLIC_KEY=$(retrieve_secret SSH_PUBLIC_KEY)
```

## Validation checklist

Before considering the app managed by this repo:

- [ ] `docs/apps.yaml` has dev and prod app entries.
- [ ] Runtime secrets are available on the correct droplets without committing secret values.
- [ ] Dev/prod DNS records point to the proxy droplet.
- [ ] Nginx routes to app droplet private IPs, not `127.0.0.1`, unless proxy and app run on the same host.
- [ ] The deploy workflow checks out the exact requested SHA/tag before building.
- [ ] Generated Compose on the droplet references the expected Docker image tag.
- [ ] `curl -fsS http://<domain>/minio/health/live` succeeds for dev and prod.

## Open architecture choices

These can be decided during implementation:

1. **Separate dev/prod image names vs. same image name with different tags**
   - Current tooling naturally uses `app.name`, which yields `restoring-warriors-images` and `restoring-warriors-images-prod` images.
   - A cleaner long-term model is one image name, `restoring-warriors-images`, promoted across environments by tag/digest.

2. **Provision-time vs. deploy-time Compose generation**
   - Provision-time generation is more declarative and keeps Ansible as the source of server state.
   - Deploy-time generation is simpler for first deploys but can drift unless carefully templated.

3. **Secret delivery mechanism**
   - Droplet-local `.env` files are simple.
   - Docker secrets are cleaner for container runtime but require more Compose/template work.
   - GitHub Actions SSH-based writes are straightforward but require careful masking and file permissions.

Recommended defaults: use one image name if the tooling is easy to adjust; otherwise accept current `app.name` image naming for the first deployment. Generate runtime Compose from this repo, keep app repo Compose local-only, and use droplet-local `.env` files with strict permissions for MinIO credentials.
